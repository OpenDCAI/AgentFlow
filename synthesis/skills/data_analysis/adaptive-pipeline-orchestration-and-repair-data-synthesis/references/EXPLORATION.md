# Phase 1: Environment Exploration Guide
* **Exploration Strategy**:
  - Decompose the task into reversible stages before writing full code. Typical stages include discovery, preparation, analysis, validation, and answer formatting, and each stage should produce an observable checkpoint. This makes it possible to revise one stage without restarting the whole workflow from scratch.
  - Use intermediate artifacts as control points. Save temporary summaries, filtered tables, or notebook outputs that reveal whether the current plan is on track. For example, a pipeline that sums fraud losses should inspect a year-filtered intermediate table before trusting the final total.
  - Treat errors and mismatches as planning signals, not just coding bugs. A failed run may indicate a wrong file choice, wrong scope, wrong rule interpretation, or wrong output contract, and the agent should revise the plan accordingly. This is especially important in real data lakes where the first hypothesis is often incomplete.
  - Keep the workflow bounded and goal-directed. Adaptive orchestration is not endless wandering; each revision should explicitly respond to an observed issue and move the pipeline closer to the answer. An edge case to avoid is uncontrolled retry loops that keep changing code without learning from feedback.
* **Target Trajectory Profile**:
  - A good trajectory should contain at least one visible planning decision and one visible repair decision. The first defines the workflow hypothesis, and the second responds to evidence from execution or inspection. Without both, the trace teaches static analysis rather than adaptive orchestration.
  - The trajectory should preserve stage boundaries. Reviewers should be able to tell when the agent was exploring, when it was executing, when it was validating, and when it revised the workflow. This staged structure is what makes the orchestration logic reusable in synthesized data.
  - The trace should end in a coherent, end-to-end artifact or answer. Even if the focus is workflow control, the final output still matters because it shows that orchestration served a concrete analytical goal. A repair-heavy trace that never converges is not a good positive example.
  - The repair step should be evidence-based. The agent should revise because of an observed mismatch, missing field, bad file, traceback, or implausible intermediate result. Random changes with no stated trigger do not provide strong supervision.

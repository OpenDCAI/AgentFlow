# Phase 2: Trajectory Selection Criteria
* **Acceptance Metrics**:
  - Accept only trajectories that identify a primary evidence table and justify why alternative candidates were rejected. The evidence table must match the requested entity type, time scope, and metric type with no unresolved ambiguity. For example, a table about school districts should be rejected if the question requires individual schools and no bridge table is found.
  - Accept only if the candidate-to-final narrowing is logically auditable. A reviewer should be able to reconstruct the ranking logic from recorded observations such as titles, columns, units, and sample rows. If the final table appears without a visible retrieval rationale, the trajectory is too brittle for synthesis.
  - Accept only if the trajectory establishes the exact answer-bearing fields before computation. That includes the target column, grouping field if needed, and any normalization such as percentages versus raw counts. An edge case is a table containing both eligible free rate and total free lunch count, where failing to distinguish them makes the trajectory invalid.
  - Accept only if the retrieval burden is non-trivial but still solvable within a small number of focused inspections. The trajectory should show real ambiguity, yet the answer should become determinate once the correct table is found. If success required broad trial-and-error across many unrelated files, the example is too noisy for stable data synthesis.
* **Rejection Criteria**:
  - Reject trajectories where the relevant table is effectively given away in the prompt or file naming. If a single file title reproduces the full user question almost verbatim, the task collapses into trivial lookup. Such cases do not genuinely test open-domain discovery.
  - Reject trajectories that rely on accidental lexical overlap without semantic validation. A table should not be accepted merely because it contains a keyword if its grain, population, or metric definition does not align. For example, county-level demographic tables should be rejected for a school-program question unless they directly encode the target rate.
  - Reject trajectories with unresolved duplicate candidates. If two tables remain equally plausible and the trace never disambiguates them through units, time coverage, or row inspection, synthesis would create ambiguous supervision. This is especially dangerous when one table is a cached summary and the other is the authoritative detail table.
  - Reject cases where the final answer requires substantial non-tabular reasoning after retrieval. This skill is about finding the right structured evidence first, not about long analytical pipelines after evidence is already known. If the main challenge is downstream modeling rather than table selection, route the example to another capability.

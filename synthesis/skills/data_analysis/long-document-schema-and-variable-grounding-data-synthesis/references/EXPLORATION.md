# Phase 1: Environment Exploration Guide
* **Exploration Strategy**:
  - Begin with documentation retrieval, not dataset scanning. First search the codebook, user guide, or methodology report for the central nouns in the question, then collect candidate variable names, weight variables, and population definitions. For example, a household-income question should trigger searches for household, income, pretax, family, person-level versus household-level units, and weighting instructions.
  - Separate semantic grounding from computation. During exploration, the goal is to map question terms to documented variables and rules rather than to immediately calculate. An edge case is when multiple columns look income-related but only one matches the question’s pretax or annualized definition in the guide.
  - Track documentation evidence at unit level. Record page ranges, line ranges, or section titles that define the variable, the eligible population, and any weighting or recoding rules. This matters because long documents often repeat related concepts, and a reusable trajectory must preserve the exact section that disambiguated the query.
  - Resolve coded fields before reading raw values. Many real datasets use abbreviations, category codes, or survey-specific labels that are uninterpretable without the documentation. For instance, a raw column name may look irrelevant until the codebook reveals that it stores household pretax income in current-year dollars.
* **Target Trajectory Profile**:
  - A good trajectory should surface the exact documentation units that define the answer-bearing variables. By the end of exploration, the trace should identify not only the dataset column but also the page or section that justifies its use. This makes the later synthesis robust and inspectable.
  - The trajectory should explicitly identify population and weighting rules where relevant. Real survey analysis is often wrong when models ignore sample design, household-versus-person distinctions, or year-specific adjustments. An edge case is a correct variable choice paired with the wrong unit of analysis, which should disqualify the trajectory.
  - The trajectory should minimize unnecessary raw-data probing before schema resolution. It is acceptable to inspect small samples to validate type and missingness, but the main work should be document-grounded. If the agent spends most of its effort scanning rows before understanding the schema, the trajectory is not a good fit for this capability.
  - The handoff to analysis should be fully specified. The final exploration state should name the target dataset, the exact variable(s), the relevant filters, and any documented weighting or transformation rules. An edge case to avoid is ending with several plausible variables and no final disambiguation.

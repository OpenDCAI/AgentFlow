# Phase 1: Environment Exploration Guide
* **Exploration Strategy**:
  - Lock down the statistical object before coding. Decide whether the task asks for a mean, median, percentage, correlation, regression score, overlap rate, or another summary, and identify the exact denominator or comparison set. This avoids a common failure where the model computes a nearby statistic that sounds reasonable but is not the requested one.
  - Confirm variable types and scale assumptions early. Many statistical procedures require numeric inputs, consistent units, or a defined significance criterion before computation. An edge case is a column stored as strings with commas or percent signs that silently breaks the analysis if not normalized first.
  - Use small sanity checks before the full computation. Inspect a few rows, check null counts, and verify that the selected variables have the expected support. This is especially important for regression or overlap tasks where an empty or malformed subset can produce deceptive outputs.
  - Separate numerical calculation from verbal interpretation. The calculation step should produce the exact statistic, and only then should the agent map it to a label such as poor fit or strong association if the task asks for one. This keeps the trajectory explicit and auditable.
* **Target Trajectory Profile**:
  - A good trajectory should contain one clearly named statistical procedure. The trace should make it obvious whether the agent performed linear regression, overlap computation, grouped summary, or another method. This is important because descriptive-statistical tasks are often easy to muddle through with vaguely similar outputs.
  - The trajectory should preserve the answer-bearing statistic directly. Even if the final answer is verbal, the underlying number should be visible in the trace. For example, a fit judgment should be backed by the actual R-squared value rather than only a qualitative label.
  - The statistical output should be interpretable without extra narrative scaffolding. A reviewer should be able to inspect the trace and see how the result answers the question. If the meaning of the computed number remains unclear, the case is not yet strong enough for synthesis.
  - The overall path should remain lightweight. Once the correct variables are in hand, the computation should be a short, well-specified analytical routine rather than a sprawling pipeline. If the trace becomes dominated by retrieval or modeling, the example belongs elsewhere.

# Phase 1: Environment Exploration Guide
* **Exploration Strategy**:
  - Clarify the prediction target and artifact contract first. Determine whether the task asks for a scalar forecast, a parameterized function, or a full submission file, and inspect the required output schema before choosing a modeling path. This prevents the common failure where the model logic is reasonable but the delivered artifact is unusable.
  - Identify feature readiness before fitting anything. Check whether the task expects raw features, engineered features, external hints, or provided model parameters to be used directly. An edge case is a prompt that looks like training but actually expects the agent to apply a supplied model equation rather than learn one.
  - Bind the modeling step to the specified evaluation metric. Whether the metric is RMSE, accuracy, execution accuracy, or a benchmark-specific function, the trajectory should reflect that the model is chosen and validated for the target metric. This keeps the capability grounded in evaluation-aware analysis.
  - Separate predictive logic from formatting logic. First ensure that the model or formula produces the correct values, then ensure that the values are written into the exact required file or scalar format. Many benchmark failures come from getting one of these two parts right but not both.
* **Target Trajectory Profile**:
  - A good trajectory should show an explicit predictive object. That may be a fitted estimator, a parameterized formula, or a submission-generation routine, but it must be clear what is doing the predicting. This distinguishes the capability from mere aggregation or descriptive statistics.
  - The trajectory should connect model choice or formula choice to the task requirements. A reviewer should be able to see why the agent used a regression, classifier, provided coefficients, or another predictive mechanism. An edge case to avoid is an arbitrary model pick with no tie to the task or metric.
  - The output artifact should be fully compliant. For file-based tasks, the trace should end with the required file name and column structure; for scalar tasks, it should end with the exact output format requested. This matters because predictive benchmarks frequently grade the artifact, not the agent’s verbal explanation.
  - The trajectory should include a lightweight validation step whenever possible. That might be checking shape, ensuring no missing predictions, or verifying a benchmark-required formatting rule. Such checks make the synthesized example much more robust for later reuse.

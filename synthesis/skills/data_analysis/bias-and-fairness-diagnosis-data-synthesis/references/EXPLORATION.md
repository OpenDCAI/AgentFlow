# Phase 1: Environment Exploration Guide
* **Exploration Strategy**:
  - First classify the bias problem before choosing a metric. Decide whether the task is about distribution bias, correlation bias, or implication bias, and whether the relevant features are numerical or categorical. For example, race and insurance type is a categorical-categorical correlation setting, not a generic fairness problem.
  - Use the method library as a routing mechanism, not just a toolbox dump. Retrieve a small set of candidate methods that match the bias type and data type, then select the most appropriate one or two for execution. This avoids the brittle habit of applying the same metric to every fairness question.
  - Preprocess with fairness sensitivity in mind. Missing values, sparse categories, and inconsistent coding can distort bias measures, so the exploration phase should standardize representations before testing. An edge case is a protected attribute with many near-duplicate string variants that would artificially dilute observed bias.
  - Plan for explanation alongside measurement. Bias-diagnosis tasks are stronger when the trajectory anticipates both the numerical result and the way it will be explained through level labels, charts, or recommendations. This keeps the task useful for both expert and non-expert users.
* **Target Trajectory Profile**:
  - A good trajectory should name both the bias type and the chosen detection method. The trace should make clear why a particular statistic or test was used for the feature pair or subgroup structure at hand. This is essential because fairness tasks are often invalidated by metric mismatch.
  - The trajectory should preserve the mapping from raw result to interpretable bias level. Whether the output is a score, effect size, or categorical severity label, the trace should show how the numerical result supports the stated conclusion. An edge case to avoid is a report that says bias exists without showing the measurement path.
  - The path should include at least one visualization or explanation planning step when the environment supports it. Bias questions are often asked by users who want interpretable feedback, not only a hidden scalar. This makes the trajectory more realistic and more valuable for synthesis.
  - The diagnostic process should remain data-bound. The agent should infer fairness patterns from the provided structured data and declared metrics rather than from general social assumptions. This keeps the capability aligned with the Data Analysis Agent definition.

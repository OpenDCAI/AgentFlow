# Phase 1: Environment Exploration Guide
* **Exploration Strategy**:
  - Map the source landscape before writing substantive code. Identify which files are structured, which are semi-structured, and which contain explanatory text that defines how the structured fields should be interpreted. For example, a payments task may need one CSV for transactions, one JSON for fees, and one Markdown file for business rules.
  - Break the question into subclaims that can each be grounded in a different source. Good exploration asks where the entity comes from, where the metric comes from, where the time filter comes from, and where any business rule comes from. An edge case is a question that appears to be about one table but actually hides a rule in documentation that changes the denominator or aggregation logic.
  - Probe joinability early. During exploration, confirm whether sources can be connected through explicit keys, shared labels, or aligned group definitions before investing in downstream computation. This prevents dead-end analyses where two seemingly relevant files cannot actually be linked.
  - Prefer intermediate artifacts that are semantically meaningful. When hopping across sources, create temporary views such as per-scheme fraud summaries or filtered gene sets rather than long opaque pipelines. This keeps the trajectory reusable and reveals where the multi-hop burden truly lies.
* **Target Trajectory Profile**:
  - A good trajectory should show at least two distinct evidence hops with a clear intermediate state between them. The first hop should resolve where each ingredient lives, and the second should combine them into a derived result. This structure is what distinguishes multi-hop reasoning from one-file analysis.
  - The trajectory should preserve source-specific assumptions. If one source defines a fraud event and another source defines the grouping entity, the trace must record both before computing rates. An edge case to avoid is silently importing a definition from prior knowledge instead of the provided context.
  - The intermediate outputs should narrow uncertainty rather than merely copy data around. Good traces produce interpretable intermediate tables, counts, or sets that make the final answer almost inevitable. If the trajectory contains many file opens but no meaningful intermediate reasoning, it is weak supervision.
  - The final answer should be short and objective even if the path is long. Multi-hop difficulty should come from source integration and chained reasoning, not from subjective report writing. This makes the resulting synthetic data easier to evaluate and reuse.

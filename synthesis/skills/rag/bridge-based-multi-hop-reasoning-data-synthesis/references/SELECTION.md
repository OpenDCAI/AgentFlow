# Phase 2: Trajectory Selection Criteria
* **Acceptance Metrics**:
  - **A visible bridge connects the support set.** Accept only if the supporting documents are linked by an identifiable bridge entity, bridge topic, or equivalent connective structure. The bridge must matter to the reasoning, not merely coexist in the documents. If removing the bridge leaves the question unchanged, the sample is too weak.
  - **At least one non-trivial hop is required.** Accept only when the answer requires crossing from one document’s claim to another through the bridge. If the answer is directly stated in a single support document, reject it. Multi-hop means the hop is indispensable.
  - **The chain is auditable step by step.** Each hop should be explainable in a short, verifier-friendly note. Reviewers must be able to reconstruct the reasoning path from the trajectory. Opaque leaps make the sample unsuitable for synthesis pipelines.
  - **The dominant difficulty is bridge traversal.** Accept when the core challenge is selecting and following the right bridge rather than heavy arithmetic, table parsing, or long-context coverage. Those other operations may appear, but the bridge should remain the main hinge. This keeps the skill well scoped.
* **Rejection Criteria**:
  - **Bridge is accidental rather than necessary.** Reject if the shared entity or topic is present but does not actually control the reasoning chain. In that case the bridge is decorative, not functional. Functional bridges are required.
  - **Too many equally plausible bridge paths.** Reject if multiple bridge paths appear equally valid and the benchmark does not clearly specify which one is intended. Such cases create annotation instability and encourage spurious reasoning. The correct path should be uniquely favored by the evidence.
  - **Single-hop shortcut exists.** Reject if one source directly answers the question without requiring the bridge at all. That shortcut would let models solve the sample without demonstrating multi-hop capability. Always test for shortcut answers before keeping the item.
  - **Bridge depends on external ontology knowledge.** Reject if recognizing the bridge requires hidden schema knowledge that is not available in the corpus or obvious from the task format. The target agent is retrieval-based, so the bridge should be inferable from the environment itself. Otherwise success becomes overly model-dependent.

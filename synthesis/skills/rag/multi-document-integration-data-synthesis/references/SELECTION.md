# Phase 2: Trajectory Selection Criteria
* **Acceptance Metrics**:
  - **No single document is sufficient.** Accept only when at least two documents are jointly necessary for the correct answer. Test this by removing each supporting document one at a time and checking that the answer becomes incomplete or impossible. This leave-one-out check is essential.
  - **Each support document contributes a unique slot.** Accept only if every kept document adds a distinct answer component or constraint. Document roles should be explainable in one short phrase such as 'launch date of API' or 'award-year anchor'. Unique contribution prevents accidental redundancy.
  - **Integration is explicit in the trace.** The accepted trajectory must show the intermediate collected facts before the final synthesis step. A hidden leap from documents to answer is not enough, even if the answer is correct. The benchmarked capability is the composition process itself.
  - **Answer form is directly checkable.** The final answer should be easy to compare against a gold target, such as an entity, tuple, short phrase, or tightly bounded sentence. This avoids subjective grading and keeps the skill reusable for data synthesis at scale. Overly open-ended answers should be reformulated or discarded.
* **Rejection Criteria**:
  - **Redundant support set.** Reject if the same answer can be recovered from a single document or from duplicated paraphrases. That would turn the sample into ordinary extraction. Multi-document integration requires true complementarity.
  - **Unclear merge rule.** Reject if the relation between the partial facts is ambiguous or if multiple valid merge rules yield different answers. The agent should not need to guess how the pieces are meant to fit together. Annotation stability matters more than surface cleverness.
  - **Support documents require another hidden capability.** Reject if the real difficulty is mostly arithmetic, table parsing, or long-chain graph traversal rather than integration of complementary facts. Those cases should be routed to more specific skills. Keep this skill focused on cross-source assembly.
  - **One support slot is too weakly grounded.** Reject if one of the partial facts depends on vague phrasing or is only inferable through unstated world knowledge. Every required slot should be explicitly recoverable from the bounded corpus. Otherwise the final answer will blur retrieval with latent memorization.

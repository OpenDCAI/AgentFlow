# Phase 2: Trajectory Selection Criteria
* **Acceptance Metrics:**
  * **Multi-source sufficiency must be provable.** Accept only if the trajectory contains every evidence piece needed for the final comparison, grouping, or chain inference. This can be checked by removing one document's evidence and asking whether the answer becomes underdetermined or wrong. If the answer survives unchanged, the trajectory is too weak.
  * **Integration must be explicit and logically correct.** The path must show the actual cross-document operation, not merely list facts and jump to a conclusion. Whether the operation is ranking, clustering, temporal trend inference, or citation chaining, the evaluator should be able to replay it deterministically. If replaying the operation yields more than one plausible answer, reject the example.
  * **Normalization must be complete.** All compared items must be aligned on entity identity, unit, time period, and criterion definition before synthesis. This protects against fake difficulty caused by sloppy annotation. A good acceptance test is whether an external reviewer could rebuild the comparison table from the trajectory alone.
  * **Non-trivial dispersion must be present.** The evidence should truly be dispersed across the corpus rather than accidentally concentrated in one document or one page. This keeps the task diagnostic for multi-document reasoning instead of single-document lookup. At least two separate documents should contribute indispensable information, and preferably more.
* **Rejection Criteria:**
  * **Reject pseudo-multi-document questions.** Discard examples where one document alone contains the final answer, even if other documents are present as decoration. Those examples test attention, not integration. The same applies when a summary page collapses all needed evidence into one place.
  * **Reject ambiguous aggregation criteria.** If the grouping rule, ranking criterion, or temporal relation is not crisply defined, the synthetic example will be hard to score and easy to dispute. A threshold like "high" or "low" must be numerically specified when used for classification. If the criterion depends on hidden domain knowledge, reject it.
  * **Reject unstable cross-document mappings.** Throw away examples where document identity, entity matching, or time alignment is so noisy that different readers would pair evidence differently. The capability should stress reasoning, not annotation chaos. An edge case is when two companies share a near-identical alias and the corpus never disambiguates them.
  * **Reject paths that skip the integration step.** If the trajectory goes directly from a few extracted snippets to a final answer without showing the comparison or chain logic, it should not pass. That makes the data unsuitable for training or evaluating reasoning behavior. A polished answer is not a substitute for an explicit reasoning trace.

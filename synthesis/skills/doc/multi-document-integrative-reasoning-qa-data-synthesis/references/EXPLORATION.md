# Phase 1: Environment Exploration Guide
* **Exploration Strategy:**
  * **Map all candidate documents before deep reading.** Start by identifying which documents are even eligible to contribute one piece of the final answer, and record their roles separately. This avoids the common failure mode where the agent over-reads one convenient document and ignores the rest. For example, if the final answer needs three yearly values, the exploration phase should explicitly identify one candidate document per year.
  * **Collect evidence in parallel, not serially.** After the candidate set is known, extract the target field or relation from each candidate document into a temporary comparison sheet. This makes cross-document integration explicit and prevents the agent from forgetting early evidence while reading later files. An edge case is when two documents use different labels for the same concept; the sheet should normalize them immediately.
  * **Preserve document identity throughout the trajectory.** Every recovered fact must retain its document source, page context, and role in the final reasoning chain. This is crucial because the skill depends on integration, not on flattening evidence into an anonymous pile. A concrete example is keeping separate columns for company name, reporting period, metric value, and source location.
  * **Probe for hidden dependencies.** Do not assume the task is mere comparison; test whether the final answer also requires ordering, grouping, citation linking, or temporal inference. This improves exploration efficiency because it tells the agent what additional evidence must be gathered before synthesis. A common edge case is a question that appears comparative but actually requires a trend statement over time.
* **Target Trajectory Profile:**
  * **A good trajectory makes every relevant document indispensable.** If one contributing document can be removed without changing the answer, the example is probably not yet sufficiently integrative. The best trajectories force the model to use all required evidence pieces. This is the defining property of strong multi-document reasoning data.
  * **A good trajectory separates retrieval from integration.** First the path should surface document-specific facts, and only after that should it perform the cross-document operation such as compare, cluster, rank, or reason through a chain. This makes the source of each final conclusion auditable. If extraction and integration are fused into one vague step, the trajectory is too opaque.
  * **A good trajectory includes an explicit normalization step.** Multi-document inputs often vary in units, labels, temporal granularity, or naming conventions, so the path should show how those differences were reconciled. This proves that the example tests reasoning rather than lucky lexical similarity. An edge case is treating quarterly and annual figures as directly comparable without normalization, which should be disallowed.
  * **A good trajectory ends in an integrated answer, not a list of fragments.** The final output should reflect a real reasoning product such as a winner, a trend, a grouped classification, or a citation chain. That is what differentiates this skill from plain extraction. A simple concatenation of facts is not enough.

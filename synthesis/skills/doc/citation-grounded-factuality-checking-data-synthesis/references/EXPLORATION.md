# Phase 1: Environment Exploration Guide
* **Exploration Strategy:**
  * **Decompose the answer into checkable claims.** Before deciding whether an output is grounded, split it into atomic or sentence-level propositions that can each be compared against the source. This reduces the chance that one unsupported clause hides inside an otherwise correct paragraph. An edge case is a response that mixes a supported fact with an unsupported causal explanation in the same sentence.
  * **Trace every claim to its cited or retrieved evidence.** For each proposition, locate the exact passage that is supposed to support it and note whether the support is direct, partial, absent, or contradictory. This turns citation checking into an explicit alignment task rather than a trust exercise. A practical example is following a legal answer’s linked case citation to see whether the stated standard actually appears there.
  * **Actively look for misgrounding, not just fabrication.** A real citation or real source passage does not guarantee valid support, so exploration should test semantic fit between claim and evidence. This matters because many dangerous hallucinations are misgrounded rather than wholly invented. A common edge case is a real authority that addresses a nearby issue but not the proposition asserted in the answer.
  * **Record support scope and missing conditions.** When evidence supports only part of a claim, note exactly what is missing, overgeneralized, or contradicted. This helps later selection distinguish partial support from full support. A strong example is a summary sentence that is correct only if one omitted condition from the source were restored.
* **Target Trajectory Profile:**
  * **A good trajectory aligns claims and evidence one by one.** The path should make it easy for a reviewer to see which source span was used to evaluate each proposition. This is essential because factuality checking is fundamentally alignment work. If the path only cites a whole document without proposition-level matching, it is too coarse.
  * **A good trajectory distinguishes support types explicitly.** Rather than collapsing everything into true or false, it should show whether the relation is supported, partially supported, contradicted, or unsupported. This makes the resulting data much more useful for both training and evaluation. An edge case is a sentence whose headline claim is supported but whose justification clause is not.
  * **A good trajectory surfaces hidden hallucination modes.** The path should include at least one opportunity for a checker to catch citation mismatch, unsupported elaboration, or contradiction that would fool a superficial reader. This creates diagnostic value. A perfectly clean claim-evidence pair is fine for balance, but not as the only pattern.
  * **A good trajectory ends in an auditable support judgment.** The final label and explanation should be reproducible from the evidence path without requiring unstated assumptions. This keeps the synthetic corpus verifiable. If two different reviewers would need to invent missing reasoning to justify the same label, the trajectory is underspecified.

# Phase 2: Trajectory Selection Criteria
* **Acceptance Metrics:**
  * **Each judged claim must have an explicit evidence target.** Accept only if the trajectory tells the evaluator exactly which passage or citation was checked for each claim. This is the minimum structure needed for support verification. Without it, the example is too vague to be useful.
  * **The support relation must be semantically decisive.** A selected example should make clear why the evidence fully supports, partly supports, contradicts, or fails to support the claim. This guards against keyword-overlap false positives. A good check is whether a human could defend the same label without consulting outside knowledge.
  * **At least one realistic failure pattern should be present.** Strong examples often include real citations that are misapplied, answers that overstate what the source says, or summaries that omit a required condition. These are more valuable than obviously fabricated nonsense. They mirror the hallucination risks found in practical legal and RAG workflows.
  * **The trajectory must remain local and inspectable.** The path should not require reviewing an entire book or entire website for one label if the decisive evidence can be localized more tightly. This keeps the benchmark practical. A good instance gives the evaluator enough evidence to judge support without drowning them in irrelevant context.
* **Rejection Criteria:**
  * **Reject evidence-free judgments.** If the trajectory or the final label depends on intuition about plausibility rather than document comparison, discard it. This skill is about grounded verification, not style critique. A fluent answer with no inspected evidence should never pass.
  * **Reject unresolvable support ambiguity.** Some claims are too vague, too broad, or too context-dependent to label reliably with the available sources. Those should be removed rather than forced into noisy annotations. A common edge case is a rhetorical summary sentence whose factual content cannot be cleanly decomposed.
  * **Reject sources that are themselves incomplete for the task.** If the benchmark instance does not actually provide the citation or grounding passage needed to decide support, discard it. Otherwise the task becomes one of missing-data inference instead of factuality checking. A checker should not be punished for absent evidence.
  * **Reject examples where all support is trivial.** If every claim-evidence pair is an easy lexical match with no room for partial support, misgrounding, or contradiction, the set will under-train the real capability. Keep some easy positives for balance, but do not let them dominate. The point is to model realistic hallucination risk.

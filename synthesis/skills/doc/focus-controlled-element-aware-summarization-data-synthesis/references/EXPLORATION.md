# Phase 1: Environment Exploration Guide
* **Exploration Strategy:**
  * **Extract the control frame before summarizing.** At the start of exploration, rewrite the task into an explicit coverage checklist such as entity/date/event/result, topic query, aspect list, or review dimension. This makes the summary target operational instead of intuitive. An edge case is a user request that sounds broad but secretly prioritizes one perspective, such as “just tell me the customer-service issues.”
  * **Search for on-frame evidence and off-frame temptations separately.** Build one ledger for content that clearly satisfies the requested focus and another ledger for attractive but irrelevant details. This separation is crucial because controllable summarization fails as much by over-inclusion as by omission. A strong example is a meeting transcript with many interesting side discussions that are not relevant to the asked topic.
  * **Use slot filling when the frame is structured.** If the task is element-aware, assign evidence to predefined slots before drafting the final summary. This helps the agent notice missing fields and keeps the eventual output compact. A common edge case is a document that richly describes events but never states the result explicitly, which should be surfaced during exploration.
  * **Check coverage balance across the requested focus.** Do not over-collect one element simply because it is easy to find. Instead, inspect whether each required slot or aspect has at least one grounded supporting span. If one focus area is weakly evidenced, note that early so the final summary can stay faithful rather than inventing balance.
* **Target Trajectory Profile:**
  * **A good trajectory mirrors the requested control structure.** The steps should be visibly organized around the user’s frame, not around the document’s incidental order. This is what distinguishes controllable summarization from generic compression. If the path cannot be mapped back to the requested focus slots, it is too loose.
  * **A good trajectory makes omission decisions explicit.** The path should show why some salient-looking details were deliberately left out because they were off-frame. This is valuable training signal because agents often over-summarize instead of obeying focus. An edge case is a dramatic anecdote that is memorable but does not help answer the meeting query or aspect request.
  * **A good trajectory demonstrates complete on-frame coverage.** Every required element, aspect, or query need should be represented by at least one grounded observation before synthesis begins. This prevents the final summary from sounding selective in an uncontrolled way. If one slot is empty, the trajectory should either reject the instance or surface the absence clearly.
  * **A good trajectory ends in an auditable controlled summary.** A reviewer should be able to take the final answer and point back to the exact frame items it satisfies. This makes evaluation cleaner and the data more reusable. A summary that is good in general but not obedient to the frame should not pass.

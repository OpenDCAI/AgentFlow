# Phase 1: Environment Exploration Guide
* **Exploration Strategy**:
  - **Separate evidence gathering from operation design.** During exploration, first identify the raw facts needed, and then explicitly name the operation required after retrieval, such as intersect, rank, sum, convert, or classify. This separation helps diagnose whether the difficulty lies in search or in post-retrieval reasoning. A clean operation label is essential for later synthesis.
  - **Prefer compact but non-trivial operations.** Choose operations that are easy for a human reviewer to verify yet still strong enough to break naive answer copying. Examples include small arithmetic, row selection, filtering by multiple conditions, or output-format conversion. Gigantic calculations are unnecessary and can obscure the RAG signal.
  - **Use structured evidence sources.** Favor tables, infobox-like layouts, report fields, or clearly formatted attribute lists whenever possible. These structures naturally create realistic post-retrieval reasoning tasks. An edge case worth keeping is when the relevant row is easy to locate only after a separate entity constraint is resolved.
  - **Check operation sensitivity.** Verify that using the wrong operation would change the answer, even if the right evidence has been retrieved. This ensures the sample measures reasoning after retrieval rather than mere evidence presence. For example, sorting versus filtering should not accidentally yield the same output.
* **Target Trajectory Profile**:
  - **Raw fact extraction before transformation.** A good trajectory first lists the raw retrieved facts or table cells and only then applies the operation. Do not fuse extraction and computation into an uncheckable sentence. Separating them makes the reasoning reproducible.
  - **Operation is named explicitly.** The trace should state the exact operation being applied, such as 'take the intersection', 'read the table cell', 'add five years', or 'convert to Roman numerals'. This teaches the downstream model to treat these operations as deliberate acts. Unnamed operations often disappear inside fluent prose.
  - **Intermediate state is retained.** Keep the intermediate result visible whenever the operation has more than one stage. For example, preserve the derived year before numeral conversion or the candidate set before final filtering. Intermediate visibility makes auditing easy.
  - **Final answer respects requested format.** The answer should match the output format demanded by the question, whether numeric, categorical, grouped JSON-like structure, or converted notation. A correct raw fact in the wrong format is still a failure for this skill. Format obedience is part of the capability.

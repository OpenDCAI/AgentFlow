# Phase 1: Environment Exploration Guide
* **Exploration Strategy**:
  - **Start from the latent decision point.** Begin by identifying the exact step where temporal tabular reasoning becomes necessary, rather than exploring the environment indiscriminately. Then enumerate the minimum schema objects, values, or metadata items that would force an agent to exercise that step in order to solve the task. For example, a good seed for this capability is an environment where a superficially plausible shortcut exists, but fails unless the agent performs the correct grounding or reasoning move.
  - **Probe the environment with low-cost structural reads before committing to full SQL construction.** Inspect schema descriptions, candidate tables, representative values, and obvious relation paths in a staged order so that the trajectory records how evidence was narrowed down. Keep each probe tied to an explicit hypothesis about the task, such as “this table may contain the governing entity” or “this field may encode the missing condition.” An edge case to include is a distractor table or column whose name looks more relevant than the true one, because such distractors are what make the trajectory valuable for synthesis.
  - **Actively search for ambiguity, not just relevance.** During exploration, log every place where everyday wording, overlapping entities, near-duplicate columns, or hidden domain rules could cause the wrong SQL to look reasonable. After detecting those ambiguity points, gather the smallest amount of evidence needed to disambiguate them in a verifiable way. A strong example is an environment where two candidate interpretations survive the first pass, but only one remains after checking values, constraints, or relation structure.
  - **Preserve a replayable evidence trail.** Every exploratory action should leave behind a concrete observation that can later be turned into a trajectory step, such as a schema snippet, a sampled value pattern, or an execution-free validation result. Write these observations in a way that explains why the next action became necessary, because the synthesis phase must later convert them into a human-readable reasoning path. An important edge case is to discard environments where the agent can jump directly from the question to the final SQL without any meaningful intermediate evidence.
* **Target Trajectory Profile**:
  - **The trajectory must expose at least one genuine decision fork.** A good trajectory for temporal tabular reasoning is one where the agent temporarily faces two or more plausible options and resolves them through evidence rather than intuition. This means the log should show why one option was retained and why the others were rejected. For instance, the trajectory should contain an observation that rules out a tempting distractor table, column, or interpretation.
  - **The trajectory must remain SQL-native even before final execution.** Environment exploration may inspect schema metadata, sampled tuples, dialect documentation, or prior validated SQL examples, but it must not rely on exporting raw data to a general-purpose scripting environment. Each step should be interpretable as something a disciplined SQL agent could do in an RDBMS-centered workflow. A negative example is a trajectory that solves the task by bulk copying data into Python and deriving the answer outside SQL.
  - **The trajectory must accumulate just enough evidence to justify the final query shape.** By the end of Phase 1, the agent should know what tables, filters, joins, or special semantics are likely required, even if the exact SQL is not yet finalized. The trajectory is weak if it contains many exploratory actions that never influence the later query. A concrete edge case is a long search history with redundant probes that add noise but do not sharpen the eventual SQL design.
  - **The trajectory must stay non-trivial but bounded.** Prefer trajectories that require multiple informative actions, yet can still be summarized as a compact sequence of observations and decisions. If the environment is so easy that only one lookup is needed, the capability signal is too weak; if it is so broad that the evidence trail becomes chaotic, the later synthesized data will be hard to verify. A good target is a trajectory where each recorded step clearly changes the agent’s belief state in a way that supports the final answer.

# Phase 2: Trajectory Selection Criteria
* **Acceptance Metrics**:
  - **Evidence-bearing progression is mandatory.** Accept a trajectory only if each major action produces a new observation that materially changes the next decision, such as eliminating a distractor, confirming a key join path, or revealing a hidden constraint. Verify this by checking that every action can be paired with an explicit downstream consequence in the final SQL plan. A trajectory that contains decorative or repetitive reads without changing the reasoning state should be rejected even if it eventually reaches the right answer.
  - **The final SQL shape must be predictable from the trajectory.** Before synthesis, confirm that the approved trajectory already implies the core relational structure of the answer: the decisive tables, the essential predicates, and any specialized semantics such as ranking, temporal ordering, geospatial filtering, or domain thresholds. This does not require the exact surface SQL text yet, but it does require a logically reconstructable plan. An edge case to reject is a trajectory whose final SQL appears correct only because the agent guessed it after weak exploration.
  - **Difficulty must come from multilingual schema grounding, not from accidental noise.** Approve the trajectory only if the decisive challenge is truly the target capability dimension rather than an unrelated obstacle like malformed data, missing permissions, or underspecified output formatting. Diagnose this by asking what single skill the successful agent primarily demonstrated. If the answer is mostly “it survived bad data hygiene” or “it guessed the output columns,” the trajectory is not a clean probe of this capability.
  - **Verification feasibility is required.** Accept only trajectories that can later be checked by another evaluator using the same schema access pattern and the same evidence trail. This means the key observations must be concrete enough that a second agent could reproduce them without relying on hidden randomness or undocumented state. A useful edge case is to reject trajectories that depend on ephemeral external context or non-repeatable sampling decisions that cannot be deterministically re-created.
* **Rejection Criteria**:
  - **Reject trivially solvable environments.** Discard trajectories where the question text directly names the exact table, column, filter, and output shape so clearly that no meaningful exploration is needed. Such cases produce shallow synthetic data that overestimates agent competence. A typical example is a question whose wording almost copies the schema labels verbatim and therefore bypasses the intended capability.
  - **Reject trajectories dominated by execution luck.** If the path repeatedly executes speculative SQL until one query happens to work, without a principled sequence of observations supporting those revisions, the trajectory should be discarded. The goal is to synthesize reasoning-rich data, not roulette-style trial and error. An edge case is a history with many near-random edits and one final lucky success, because it is hard to extract a trustworthy reasoning chain from that behavior.
  - **Reject capability leakage across boundaries.** If the trajectory depends on non-SQL tooling or on a qualitatively different capability than the one being synthesized, do not keep it for this skill. For example, a trajectory meant to test dialect adaptation should not be accepted if the real bottleneck was open-domain table retrieval. Such leakage makes the resulting synthetic data harder to interpret and weaker as targeted supervision.
  - **Reject unverifiable ambiguity.** Some environments contain ambiguity that cannot be resolved from the available schema, values, or approved context. When the agent succeeds only by assuming hidden business rules or by silently inventing a mapping, the trajectory must be rejected. A concrete edge case is a task where two candidate answers remain equally plausible after all available evidence has been inspected.

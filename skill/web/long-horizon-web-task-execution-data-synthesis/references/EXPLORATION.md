# Phase 1: Environment Exploration Guide
* **Exploration Strategy**:
  - Break the task into ordered subgoals before acting. The agent should identify prerequisite steps, state transitions, and success checks so it does not forget constraints halfway through the workflow. For example, a filter-heavy shopping task naturally decomposes into sorting, width constraint, price constraint, rating constraint, and final verification.
  - Carry forward unresolved constraints explicitly after each action. A long-horizon task fails when earlier requirements silently drop out of the agent’s working state, so the trajectory should maintain a compact checklist or plan state. An edge case is a page refresh or navigation jump that visually hides earlier filter chips; the agent should still remember them.
  - Verify intermediate state rather than assuming actions succeeded. After clicking, typing, or applying a filter, inspect the page for confirmation such as changed results, selected chips, active sort controls, or newly available buttons. This protects against silent failures and accidental action on the wrong element.
  - Recover from omissions and navigation drift as first-class events. If the agent notices that a required step was missed, it should deliberately return to the relevant control and repair the state instead of pushing ahead. A good edge case is finding a product that matches price and rating but forgetting to sort by newest, which should trigger a correction rather than a premature answer.

* **Target Trajectory Profile**:
  - A target trajectory should involve a genuinely multi-step workflow, ideally six or more user-level actions or several logically distinct subgoals. The point is to stress state maintenance and completion discipline. If two or three actions are enough, the example is probably too short.
  - The trajectory should include at least one page transition, panel change, or workflow phase shift. Long-horizon browser tasks are difficult because the agent must persist across changing screens and interaction modes. A path that never changes mode is less diagnostic.
  - A strong trajectory preserves a full conjunction of requirements across the entire interaction. The agent must not only find something relevant but also keep checking that all constraints still hold after later actions. As an edge case, a candidate that satisfies most but not all conditions should remain rejected even if it appears visually attractive.
  - The endpoint should represent completed task execution rather than partial exploration. The agent must either reach the final page state, perform the requested action, or return a fully verified result after all required steps. Trajectories that stop at a promising intermediate screen are incomplete.

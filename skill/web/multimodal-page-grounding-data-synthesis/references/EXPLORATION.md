# Phase 1: Environment Exploration Guide
* **Exploration Strategy**:
  - Inspect the page through both structural and visual channels before acting. The agent should read DOM or HTML references to understand what elements exist, then check the screenshot to determine which elements are actually visible, active, or contextually relevant right now. As an edge case, if the DOM contains hidden or repeated elements, the screenshot should determine which one is action-worthy.
  - Track state transitions after every action. A multimodal task is often difficult because a click changes what is visible, what is focused, or which controls are enabled, and these changes matter to the next step. For example, opening an email exposes forward controls that were not actionable from the inbox list view.
  - Use visual order and spatial context when the task references rank, position, or local grouping. Phrases like “the 7th result,” “the left button,” or “the visible calendar slot” require the agent to reason over what is shown on screen rather than over arbitrary DOM order. If text and visual order disagree, the visible user-facing order should dominate.
  - Probe for perceptual ambiguity before committing. If two elements have similar labels, the agent should use screenshot context, surrounding text, or recent state changes to decide which element is intended. A good edge case is a page with repeated “Search” or “Submit” controls where only one lies inside the relevant panel.

* **Target Trajectory Profile**:
  - A target trajectory should contain at least one action that depends on visible page state rather than static text alone. This may involve choosing an element by on-screen order, resolving a repeated label using layout, or reacting to a newly revealed control after navigation. Without such a step, the example reduces to DOM lookup.
  - The trajectory should include at least one state transition that changes what the agent must attend to. A before-and-after shift in screenshot content is important because it forces grounded re-perception. If the entire task occurs on one unchanged screen, the multimodal pressure is weaker.
  - A strong trajectory uses HTML or DOM structure to locate candidates and visual evidence to select the correct one. This division of labor is essential for web agents, which often know many elements exist but still need grounding to choose correctly. As an edge case, two DOM nodes may share similar attributes, and only the screenshot clarifies which is visible or relevant.
  - The final success condition should be tied to interface completion, not only to information retrieval. The agent should end in a new page state, submitted action, or visibly changed UI that confirms the task is complete. If the task ends with extracted text only, it probably belongs to search reasoning rather than page grounding.

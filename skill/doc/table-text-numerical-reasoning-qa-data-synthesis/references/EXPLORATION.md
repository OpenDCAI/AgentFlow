# Phase 1: Environment Exploration Guide
* **Exploration Strategy:**
  * **Separate numeric discovery from numeric reasoning.** First locate every candidate number, unit, header, and entity binding relevant to the question before performing any arithmetic. This prevents the agent from computing over incompletely grounded values. An edge case is when the visually largest number on the page is not the requested metric at all.
  * **Read the prose around the table.** Notes, footnotes, slide captions, and narrative discussion often define the unit, reporting period, or exception that makes the table interpretable. Ignoring those textual cues is one of the fastest ways to compute the wrong answer from the right table. A concrete example is a metric reported in thousands in the table but interpreted as millions only after a footnote check.
  * **Normalize entities and periods early.** As soon as a number is found, attach it to a normalized entity, time period, and metric name in a scratch ledger. This ensures later arithmetic is done over comparable quantities instead of accidentally mixing rows or years. A strong edge case is when current-quarter and full-year numbers appear on the same page under visually similar headers.
  * **Inspect visual structure when text alone is insufficient.** For charts, dashboards, and slides, use bar heights, legends, axis labels, color mapping, and spatial grouping as first-class evidence rather than as decoration. This matters because a visually rich document can encode the answer in layout relations rather than plain OCR text. A common example is needing the highest and lowest product bar before computing a difference.
* **Target Trajectory Profile:**
  * **A good trajectory shows every numeric binding step.** The path should make clear where each number came from, what it represents, and why it is part of the final computation. This is essential for auditability because quantitative tasks fail silently when one binding is wrong. If a number appears in the answer without a source location and label, the trajectory is too weak.
  * **A good trajectory includes a unit and period sanity check.** Before arithmetic, the path should verify that all values are in compatible units and aligned to the intended time scope. This extra step is necessary because many realistic document tasks are deceptive precisely at this point. An edge case is converting or rejecting a value after discovering that the source uses a different scale.
  * **A good trajectory makes the operator explicit.** The evaluator should be able to tell whether the task required subtraction, ratio calculation, trend inference, extremum selection, or threshold comparison before reading the final answer. This reduces ambiguity and prevents the model from masking a lucky guess as reasoning. If the operator cannot be identified, the example should not be used.
  * **A good trajectory ends in a computed or inferred result, not a copied cell.** The final answer should demonstrate a genuine reasoning outcome such as a difference, percentage, ranked winner, or trend statement. That is what distinguishes this skill from evidence-localized extraction. A copied raw cell is insufficient unless the question intentionally targets direct numeric lookup.

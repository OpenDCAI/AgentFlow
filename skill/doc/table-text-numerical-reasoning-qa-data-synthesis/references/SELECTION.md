# Phase 2: Trajectory Selection Criteria
* **Acceptance Metrics:**
  * **All operands must be grounded.** Accept only if every number used in the answer is tied to a visible document source and a normalized metric label. This is the minimum requirement for reliable quantitative data synthesis. If one operand is implicit, remembered, or hand-waved, reject the trajectory.
  * **The arithmetic or comparison must be reproducible.** A reviewer should be able to recompute the final answer from the trajectory without opening hidden context or making extra assumptions. This keeps the benchmark transparent and reusable. If recomputation yields multiple plausible outputs, the example is underspecified.
  * **The example must mix retrieval with reasoning.** Accept only if the task requires both finding the right values and doing a non-trivial operation on them. This prevents the skill from collapsing into pure extraction or pure math detached from documents. A good example is asking for a difference between two visually grounded values from separate parts of a slide deck.
  * **At least one realistic numeric hazard must be present.** The trajectory should contain a point where the agent had to guard against a typical failure such as unit mismatch, label confusion, period confusion, or visual misreading. This makes the example diagnostic. If the path is numerically sterile, the task is too easy.
* **Rejection Criteria:**
  * **Reject copy-only questions.** Discard any example where the answer is just one table cell with no need for arithmetic, normalization, or cross-field reasoning. Those tasks belong to simpler grounded extraction. They are useful elsewhere but not as the core of this capability.
  * **Reject under-labeled numbers.** If the source does not clearly tell the agent what a number refers to, or if the annotation never pins that relation down, the example should be thrown away. Quantitative ambiguity is not the same as quantitative difficulty. A common edge case is two nearby rows with almost identical names.
  * **Reject mixed units unless normalization is documented.** It is acceptable to use differently scaled values only when the trajectory explicitly shows how they were converted or why they were not comparable. Otherwise the benchmark rewards annotation mistakes. Silent unit conversion is grounds for rejection.
  * **Reject unreadable visuals or corrupted tables.** Visual-document numeric reasoning is valuable only when the chart, table, or layout is independently interpretable. If axis labels are missing, OCR is catastrophically broken, or the screenshot is too blurry, discard the instance. The capability should test reasoning, not document damage.

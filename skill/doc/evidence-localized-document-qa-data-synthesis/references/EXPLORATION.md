# Phase 1: Environment Exploration Guide
* **Exploration Strategy:**
  * **Map the workspace before extracting anything.** Start by surveying headings, section labels, page thumbnails, captions, repeated headers, and other navigational cues that can reveal where a page, paragraph, table cell, chart mark, or layout region is likely to appear. Then write down a lightweight search plan instead of reading every document linearly, because this capability depends on controlled navigation before deep reading. For example, if several annual reports repeat the same metric family, the agent should first identify which report-year combinations are plausible candidates and only then zoom into the exact region.
  * **Use progressive narrowing rather than one-shot reading.** Begin with broad candidate discovery, continue with local inspection of a few promising regions, and only after that commit to a final evidence set. This forces the trajectory to expose how the answer was found instead of hiding the critical step inside an untraceable leap. An edge case is when an early candidate looks semantically perfect but later inspection shows that the number, entity, or citation belongs to another section.
  * **Track evidence as an explicit ledger.** After each exploration step, record which document, page, section, or span was inspected and what tentative fact was recovered from it. This prevents the agent from blending nearby evidence and makes it possible to reject paths that rely on memory-like paraphrase instead of grounded inspection. A concrete example is marking one table cell as provisional evidence while flagging a neighboring footnote as a possible unit-conversion risk.
  * **Actively seek discriminative cues, not just supportive cues.** During exploration, look for the signals that separate the true answer from repeated values, similar headings, nearby tables, or visually similar figures. That means checking entity identity, time period, unit, citation direction, page function, and whether the local context truly matches the question intent. A good edge case is a visually similar chart or paragraph that shares vocabulary with the question but answers a different subquestion.
* **Target Trajectory Profile:**
  * **A good trajectory exposes the dependency structure.** The final answer should be impossible to justify without the exploration steps that surfaced the right evidence unit. Each intermediate observation must move the agent closer to the answer by shrinking uncertainty rather than by restating the question in different words. If removing one exploration step leaves the answer equally likely, the trajectory is too weak for synthesis.
  * **A good trajectory is evidence-first and conclusion-second.** The answer should appear only after the decisive evidence has been located, normalized, and checked against the question constraints. This ordering matters because otherwise the agent can hallucinate a likely answer and backfill evidence afterwards. A strong example is delaying the final response until the correct year, entity, and page have all been individually confirmed.
  * **A good trajectory contains at least one non-trivial disambiguation event.** There should be a moment where the agent rejects an attractive but wrong candidate because of a concrete mismatch such as date, unit, entity, page role, or support relation. This proves that the data point is diagnostic rather than trivial. If every inspected candidate is equally valid, the synthetic example will be too easy.
  * **A good trajectory ends in a short grounded answer tied to one local evidence unit.** The trajectory should naturally terminate in a response whose wording can be traced back to the inspected evidence without broad paraphrastic drift. That makes the example auditable and robust to evaluation. An edge case is a visual chart question where the answer must still be reducible to one clearly identified plotted value.

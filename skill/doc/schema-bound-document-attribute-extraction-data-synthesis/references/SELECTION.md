# Phase 2: Trajectory Selection Criteria
* **Acceptance Metrics:**
  * **Schema fidelity must be testable.** Accept only if the trajectory makes it possible to verify that the chosen label or field satisfies the written schema definition, not just common-sense interpretation. This is the first quality gate. If the mapping depends on intuition rather than rules, reject it.
  * **Each extracted item needs a localized support span.** A selected example should include one or more tight evidence spans that justify the output. This keeps the data auditable and discourages whole-document guesswork. If the support is only “somewhere in the note,” the example is too weak.
  * **The trajectory should separate category and subtype when the schema does.** For tasks with broad categories and finer attributes, the path should show both decisions in order. This yields better supervision and cleaner error analysis. A good example first marks transportation, then marks it as adverse because it blocks care access.
  * **Borderline contrast should be present somewhere in the set.** Strong data for this skill includes not only easy positives but also neutral mentions, non-adverse mentions, and near-miss negatives. This helps train robust schema obedience. If the set only contains obvious positives, it will not generalize well.
* **Rejection Criteria:**
  * **Reject label assignments without rule evidence.** If the output label sounds right but no explicit schema criterion is satisfied in the source, discard the instance. This keeps the benchmark from drifting into fuzzy semantic tagging. A persuasive narrative is not enough.
  * **Reject sources that support too many incompatible labels at once without disambiguation.** Ambiguous multi-label cases are acceptable only if the trajectory shows how each label was justified or ruled out. Otherwise the example becomes annotation noise. A common edge case is a sentence that mentions family, work, and travel but only one is actually actionable under the schema.
  * **Reject extraction tasks that are really open-ended QA.** If the output is a paragraph answer rather than a normalized field or label, the example belongs elsewhere. This skill is specifically about schema-bound structuring. A narrative explanation may accompany the result, but it cannot be the result itself.
  * **Reject underspecified schemas.** If the rules do not make it clear what counts as positive, adverse, subtype-specific, or extractable, do not synthesize data from them. The taxonomy must stay crisp. Otherwise different annotators will silently create incompatible training targets.

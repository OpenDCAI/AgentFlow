# Phase 1: Environment Exploration Guide
* **Exploration Strategy:**
  * **Start from the schema, not the sentence.** Before reading the document deeply, rewrite the label inventory, subtype rules, and adverse-or-not criteria into an explicit checklist. This prevents the agent from drifting into intuitive semantics that do not match the annotation policy. An edge case is a sentence that feels negative in tone but does not satisfy the schema’s adverse definition.
  * **Collect positive and borderline evidence separately.** During exploration, mark spans that clearly satisfy a field definition and also spans that are close but not fully qualifying. This is important because extraction quality depends on handling near-misses correctly. A practical example is distinguishing “married” from “separated” when only one of them maps to an adverse relationship label.
  * **Preserve subtype information at the point of discovery.** If the schema has attributes such as employment type, transportation issue type, or entity subtype, capture that detail immediately instead of trying to infer it later. This keeps the trajectory faithful to the source. A common edge case is a phrase that signals both the broad category and the adverse subtype in one compact sentence.
  * **Track one supporting span per extracted item.** The agent should not output a label or field without a concrete local justification from the document. This makes the final extraction auditable and simplifies error analysis. A strong example is linking “missed appointment because her sister couldn't drive her” directly to the transportation-adverse label.
* **Target Trajectory Profile:**
  * **A good trajectory mirrors the schema hierarchy.** It should first determine whether the broad category is present and then decide subtype, status, or severity if the schema requires it. This ordering reduces confusion and makes the path easy to review. If the trajectory jumps directly to a fine-grained label without showing category membership, it is too opaque.
  * **A good trajectory distinguishes mention from condition.** Many schemas require separating whether something is mentioned at all from whether it is adverse, actionable, or eligible for extraction. The path should therefore show both decisions when needed. This is crucial in document domains where the same category can appear in both neutral and problematic forms.
  * **A good trajectory handles borderline mentions explicitly.** The best examples include at least one point where the agent considers a near-miss and rejects or downgrades it based on the rules. This teaches the model not to over-label. If every example is an easy keyword match, the skill becomes shallow.
  * **A good trajectory ends in normalized structured output.** The final answer should be a label, field set, or schema-conforming record rather than a prose paraphrase. This is what makes the example operational for downstream workflows. A free-form explanation can be added, but it should never replace the normalized output.

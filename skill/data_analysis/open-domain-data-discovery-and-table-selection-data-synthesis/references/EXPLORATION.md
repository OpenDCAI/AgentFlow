# Phase 1: Environment Exploration Guide
* **Exploration Strategy**:
  - Start by converting the user request into a retrieval sketch that names entities, measures, time windows, and possible domain synonyms. Then scan table titles, column names, short previews, and nearby metadata rather than opening every file in full. For example, a request mentioning county-level student rates should expand into county, district, school, free-rate, lunch-program, and eligibility variants before ranking candidates.
  - Prefer a widening-then-narrowing retrieval pattern instead of an immediate exact-match search. First collect a small candidate set of tables that appear semantically plausible, then compare them on grain, units, time coverage, and entity overlap. An edge case is when the table with the best title is actually a summary page while the needed value is in a lower-ranked detail table.
  - Inspect schema before content-heavy execution. Read enough rows to determine whether the table uses direct labels, coded values, or derived views, and only then commit to heavy processing. For instance, a table that looks relevant may hold only percentages by state while the question requires county-level school rows.
  - Actively test whether multiple tables are complements rather than competitors. Some environments separate metrics, entity lookup keys, and calendar filters across different artifacts, so the retrieval phase should identify minimal supporting pairs or triples. A common edge case is a question whose target table contains the metric but requires a separate reference table to resolve county or program labels.
* **Target Trajectory Profile**:
  - A good trajectory should show explicit candidate comparison rather than a lucky first hit. The trace should record at least one rejected near-miss table and the reason it was rejected, such as wrong geography, wrong year, or wrong aggregation grain. This makes the trajectory reusable for synthesis because it reveals the retrieval ambiguity that the final question depends on.
  - The trajectory should isolate a minimal evidence set before any serious computation begins. In practice that means the final answer should be traceable to one primary table or a very small set of tightly connected tables, not a diffuse crawl across unrelated files. An edge case is when the agent keeps opening new tables after already finding sufficient evidence, which usually signals weak grounding.
  - The retrieval path should preserve semantic justifications in natural language. It should be possible to read the trajectory and see why a table was chosen in terms of the user’s wording, not only because a column name happened to match. For example, the agent should explain that a table is relevant because it contains county-level K-12 eligibility rates rather than simply because it includes the word school.
  - The trajectory should leave a clear handoff point to downstream analysis. By the end of exploration, the relevant table names, join keys if any, target columns, time filters, and units should already be identified. An edge case to avoid is a trajectory that ends with “this looks relevant” but still leaves the actual metric undefined.

# Phase 2: Trajectory Selection Criteria
* **Acceptance Metrics**:
  - Accept only if the trajectory identifies the variable definition from documentation rather than guessing from column names. There must be explicit evidence that the agent read the relevant guide, codebook, or manual section. A lucky column-name match without documentary support is insufficient.
  - Accept only if any necessary survey-design or schema rules are resolved before computation. That includes weights, denominator definitions, year alignment, and entity level. For example, a household statistic computed at the person level should cause rejection even if the raw numeric result looks plausible.
  - Accept only if the documentation burden is material. The case should require consulting long or heterogeneous documents, not a one-line README with self-explanatory columns. The point is to create training data where document navigation is the core bottleneck.
  - Accept only if the final variable grounding is stable under minor wording changes. If slightly rephrasing the question would send a competent analyst to the same documented fields, the trajectory is suitable. If the mapping depends on fragile wording or undocumented assumptions, reject it.
* **Rejection Criteria**:
  - Reject cases where the columns are already self-explanatory and no long-document reasoning is needed. If the dataset clearly contains a column named exactly median_household_income_2024, there is no schema-grounding challenge. Such examples do not exercise the intended capability.
  - Reject trajectories that read documentation but fail to tie it to concrete variables. A long search trace with many copied passages is not enough if it never commits to the field names, filters, and units needed for code. This would yield weak supervision for synthesis.
  - Reject cases where the relevant documentation is contradictory or underspecified. If the guide and codebook point to conflicting definitions with no authoritative resolution, the final data point will be unstable. It is better to discard such traces than synthesize ambiguous training data.
  - Reject examples whose real difficulty lies in downstream modeling rather than document grounding. If the variable mapping is trivial and the main work is regression, optimization, or visualization, the example belongs under another capability. The trajectory should make documentation navigation the dominant challenge.

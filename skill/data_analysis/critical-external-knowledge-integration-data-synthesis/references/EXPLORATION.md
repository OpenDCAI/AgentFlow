# Phase 1: Environment Exploration Guide
* **Exploration Strategy**:
  - Treat every external hint as a hypothesis, not as a command. During exploration, classify each hint as beneficial, harmful, unverifiable, or irrelevant based on the data schema and task objective. For example, a hint that says remove Location should be challenged if location is clearly central to the target behavior.
  - Cross-check guidance against train/test structure and evaluation logic. Helpful advice should improve feature engineering, cleaning, or model choice without violating split boundaries or leaking future information. An edge case is a hint that sounds expert but would apply inconsistent preprocessing across folds or train/test partitions.
  - Look for contradiction signals among the hints. Mixed-information environments are most informative when some hints directly compete, such as add School Quality versus drop a strong proxy feature. This lets the agent demonstrate selective adoption rather than passive compliance.
  - Record explicit accept-or-reject decisions in the trajectory. A good exploration trace should say which hints were adopted, which were discarded, and why. This is crucial because the learning value comes from the agent’s judgment process, not just the final model script.
* **Target Trajectory Profile**:
  - A good trajectory should contain at least one accepted hint and one rejected hint. This demonstrates critical filtering rather than blanket trust or blanket skepticism. If the trace uses all hints or rejects all hints without analysis, it is not a strong example.
  - The trajectory should tie each adoption decision to observable data properties. Accepted hints should align with feature semantics, target type, split logic, or documented evaluation metrics. An edge case to avoid is accepting a hint purely because it sounds domain-expert-like without any schema-level validation.
  - The final pipeline should remain end-to-end executable. Even though the focus is judgment, the trace should still end in a working script or valid output artifact such as submission.csv. This keeps the capability grounded in practical data analysis rather than abstract debate.
  - The reasoning should expose why adversarial guidance was harmful. Good traces show concrete failure modes such as dropping an informative variable, mishandling time, or introducing leakage. This makes the synthesized example useful for training robust analytical agents.

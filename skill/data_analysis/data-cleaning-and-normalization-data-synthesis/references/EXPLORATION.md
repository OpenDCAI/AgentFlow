# Phase 1: Environment Exploration Guide
* **Exploration Strategy**:
  - Inspect data quality markers before choosing a cleaning action. Check column types, missingness, range anomalies, formatting consistency, and whether the prompt specifies a particular preprocessing rule. For example, a Z-score outlier task requires verifying that the target column is numeric before any standardization is attempted.
  - Prefer rule-bound cleaning over free-form improvisation. If the task specifies a method such as Z-score, min-max normalization, or row/column completeness, the agent should align exactly with that rule instead of inventing a different preprocessing strategy. An edge case is when a model uses IQR logic for an explicitly Z-score-based question, which must be rejected.
  - Separate diagnostic inspection from mutating transforms. First determine what is wrong or what needs normalization, then apply the documented transform in a controlled way. This matters because many data-cleaning failures come from silent type coercions or hidden row drops.
  - Track the effect of cleaning in a compact audit trail. A strong trajectory records how many rows, cells, or values were affected and why. This becomes especially useful when the cleaned output feeds a later statistical or modeling step.
* **Target Trajectory Profile**:
  - A good trajectory should reveal the exact preprocessing rule and its execution order. The trace should show that the agent understood the data issue, applied the right transformation, and only then computed the requested output. This is important because cleaning tasks are often invalidated by doing the right operations in the wrong order.
  - The trajectory should preserve analytical fidelity to the prompt. If the question binds the method, threshold, or expected post-cleaning shape, the agent must follow those constraints exactly. An edge case to avoid is a trajectory that cleans the data plausibly but not in the way the task required.
  - The cleaning effect should be measurable. By the end of exploration and execution, the trace should support a concrete count, flag set, normalized field, or cleaned shape. This makes the case easy to verify and strong for later synthesis.
  - The trajectory should avoid unnecessary modeling or reporting overhead. The essence of this capability is accurate preprocessing, not elaborate downstream interpretation. If the main work comes after cleaning, the case may belong under another capability.

# Phase 1: Environment Exploration Guide
* **Exploration Strategy**:
  - Search for rule-bearing documents before doing any main computation. Domain-rule tasks often hide the decisive logic in manuals, fee guides, codebooks, or compliance notes rather than in the data table itself. For example, a cost-effectiveness question over authorization indicators is unsound unless the indicator semantics are first grounded in documentation.
  - Classify each rule as explicit, implicit, or composite. An explicit rule directly states the needed formula, an implicit rule requires interpretation across several sentences, and a composite rule links multiple clauses that must be applied together. This classification helps the agent decide whether a simple lookup or a more careful synthesis step is needed.
  - Translate narrative rules into executable conditions as early as possible. Write the rule in operational terms such as filters, mappings, piecewise formulas, or scenario adjustments before touching the full dataset. An edge case is when a seemingly descriptive paragraph actually changes the denominator or the applicable subgroup.
  - Validate rule scope against the data. A rule may apply only to a year range, merchant type, fee class, or authorization path, so the exploration phase must confirm the scope before applying it globally. This prevents silent over-application of domain logic.
* **Target Trajectory Profile**:
  - A good trajectory should include an explicit rule extraction moment. The trace should show where the agent read the contextual instruction, how it paraphrased it into operational logic, and how that logic changed the subsequent computation. This is the core evidence that the capability was exercised.
  - The trajectory should make it obvious that data alone was insufficient. A reviewer should be able to see that the raw fields could not have produced the correct answer without the contextual rule. If the rule never changes the computation, the example is not strong enough.
  - The rule application should be scoped and auditable. The trace should specify which subset of rows, categories, or scenarios the rule was applied to and why. An edge case to avoid is a trajectory that uses a correct rule but applies it to all data indiscriminately.
  - The final answer should remain crisp even when rule extraction is complex. Good domain-rule tasks end in a value, option, or concise scenario output, making correctness easy to evaluate. This keeps the difficulty concentrated in rule grounding rather than in subjective narration.
